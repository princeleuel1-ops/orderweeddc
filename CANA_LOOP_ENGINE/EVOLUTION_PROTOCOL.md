# CANA Evolution Protocol

## Objective

Keep converting repository evidence into bounded, measurable product
improvements for the persisted execution window. The runtime must remain alive
through provider waits and crashes, but it must not call waiting, repeated
analysis, or model agreement “progress.”

No finite runtime can guarantee “1000x,” singularity, or industry dominance.
It can guarantee that every accepted change is tied to a baseline, a falsifiable
hypothesis, direct feedback, independent criticism, deterministic verification,
and rollback.

## Fundamental loop

1. **Sense** — fingerprint repository, control-tower, test, browser, HTTP,
   security, performance, and provider state.
2. **Select** — identify one specific SOTA gap and rank it against alternatives.
3. **Model** — list assumptions and name exactly one weakest brittle point.
4. **Act** — implement one bounded strategy revision in an isolated worktree.
5. **Feel** — collect machine feedback from the named success metrics. This is
   the software feedback harness; no physical haptic device is implied.
6. **Criticize** — independent roles try to falsify the evidence and brittle
   point.
7. **Verify** — deterministic commands run against the exact candidate.
8. **Integrate or reject** — accepted work integrates reversibly; failure
   preserves evidence and changes strategy.
9. **Measure again** — post-integration checks must prove the delta survived.

## Compounding frontier contract

Every newly selected high-impact mission must include current comparison
evidence, a numeric baseline, a strictly better outcome target, a numeric
non-regressing guardrail, a falsification test, an exact promotion rule, and
the next harder frontier. These fields are persisted, exposed in sanitized
status, and delivered unchanged to the builder and reviewers.

The frontier epoch advances only after a measured modifying candidate
integrates. Pending candidates, model agreement, elapsed time, token use, and
file count do not advance it. If an outcome misses or a guardrail regresses,
the exact candidate is rejected and the failure becomes strategy evidence.

After a verified promotion, the observed result becomes the next baseline.
The loop then attacks the next highest-value constraint; it does not claim that
the product is globally or permanently “best.”

## Invariants

- Credentials, authorization, truthful-data, provenance, and test protections
  never weaken to increase velocity.
- A model cannot approve, merge, or release its own work.
- Modifying jobs use isolated worktrees and immutable candidate revisions.
- Integration requires independent criticism, deterministic evidence, release
  judgment, a clean primary tree, and rollback.
- Public deployment, force push, credential operations, purchases, and legal or
  business determinations remain outside autonomous authority.
- PID existence is not process identity.
- Idle time, token use, file count, and cycle count are not success metrics.

## Required mission contract

Every mission persists:

- assumptions;
- one SOTA gap;
- one brittle point;
- baseline and success metrics;
- feedback signals;
- acceptance criteria and prohibited actions;
- strategy revision;
- measured progress delta.
- current benchmark evidence and observation time;
- numeric outcome and guardrail measurement contract;
- falsification test, promotion rule, next frontier, and frontier epoch.

Missing fields fail closed. Selection-only Codex work must emit a strict
`CANA_JSON_START` / `CANA_JSON_END` contract. The supervisor validates and
materializes it into a bounded builder mission. Malformed selection output
stops after a configured threshold instead of consuming capacity forever.

## No-progress rule

The approach fingerprint uses the stable plan and strategy revision, failure
class, changed files, test failure, criteria movement, score movement, and
reversion. Session IDs, attempt-specific prompt text, and output hashes do not
reset it.

At the threshold:

- the repeated approach becomes terminal;
- a child strategy revision is queued;
- routing advances to a fallback when available;
- the brittle point records why the prior strategy failed;
- unrelated workstreams remain schedulable.

## Capacity and continuity

`daily_job_ceiling: 0` delegates exhaustion to the actual provider instead of
stopping after an arbitrary local count. Real usage limits, authentication,
timeouts, and cooldowns remain fail-closed and durable.

External review blocks integration, not invention. Up to
`maximum_pending_review_candidates` high-impact candidates may be built
sequentially in isolated worktrees while earlier candidates await independent
review. The cap prevents an unbounded speculative backlog, and every candidate
still requires exact-hash review and deterministic gates before integration.

Before any worktree or model call, the stored mission baseline must equal the
current protected integration base. For a high-impact recommendation, its
parent selection baseline must match too. The same comparison runs again when
selection output is accepted, forming a compare-and-swap boundary around
inference. A mismatch is rejected as stale without usage, and exactly one fresh
selection is scheduled from current evidence.

The builder backlog is bounded. When no evidence changed and no executable
mission is available, heartbeat state is `idle:no-evidence-delta`. The
supervisor stays recoverable without fabricating improvement.
